6. Configuration

Dalam bab ini, adalah dijelaskan mengenai objek Kubernetes yang berhubungan dengan konfigurasi. Konfigurasi digunakan untuk menambahkan environment variable atau berkas konfigurasi ke dalam container di dalam pod sehingga aplikasi yang dijalankan di dalam container dapat dikonfigurasikan tanpa harus repot-repot mengubah command dari container.

Selain untuk mengonfigurasi container dalam pod, akan dijelaskan juga objek Kubernetes yang mengonfigurasi resource dari suatu namespace sehingga pengguna dapat membatasi atau memberikan nilai default dari resource yang digunakan di dalam suatu namespace.

ConfigMap

ConfigMap
adalah suatu objek berupa kumpulan data dalam format key value. Configmap berguna untuk menyimpan data konfigurasi non rahasia dari aplikasi terutama aplikasi dalam container.

Untuk melihat daftar configmap yang ada, dapat menggunakan perintah berikut

xxxxxxxxxx
kubectl get configmap --all-namespaces

Ganti --all-namespaces menjadi --namespace=<namespace> untuk melihat daftar configmap dalam suatu namespace.

Untuk membuat sebuah configmap, secara sederhana adalah menggunakan perintah berikut

xxxxxxxxxx
kubectl create configmap <configmap_name> [--namespace <namespace>] [--from-literal=<key>=<value>] [--from-file=[<key>=]<filename>] [--from-file=<directory>] [--from-env-file=<envfilename>]

Untuk menghapus configmap, dapat menggunakan perintah berikut

xxxxxxxxxx
kubectl delete configmap <configmap_name> [--namespace <namespace>]

Berikut adalah struktur berkas yaml dari configmap

xxxxxxxxxx
apiVersion: v1
kind: ConfigMap
immutable: <immutable: boolean>
metadata:
  name: <configmap_name: string>
  namespace: <namespace: string>
  labels:
    <label_key: string>: <label_value: string>
  annotations:
    <annotation_key: string>: <annotation_value: string>
binaryData:
  <binary_data_key: string>: <binary_value>
data:
  <data_key: string>: <data_value>

key dalam binaryData dan data tidak boleh ada yang sama

Configmap dapat dimasukkan ke dalam container sebagai environment variable baik seluruh data maupun beberapa. Untuk memasukkan suatu data dari configmap ke dalam container, gunakan parameter berikut di dalam struktur dari container

xxxxxxxxxx
env:
  - name: <env_name: string>
    valueFrom:
      configMapKeyRef:
        key: <data_key: string>
        name: <configmap_name: string>
        optional: <optional: boolean>

Untuk memasukkan seluruh data dari configmap ke dalam container, gunakan parameter berikut di dalam struktur dari container

xxxxxxxxxx
envFrom:
  - prefix: <prefix: string>
    configMapRef:
      name: <configmap_name: string>
      optional: <optional: boolean>

Berikut adalah contoh berkas yaml dari configmap

xxxxxxxxxx
apiVersion: v1
kind: ConfigMap
metadata:
  name: webconfig
  namespace: default
data:
  port: 80
  username.default: admin

Secret

Secret
adalah suatu objek berupa kumpulan data dalam format key value. Secret berguna untuk menyimpan data konfigurasi rahasia dari aplikasi terutama aplikasi dalam container.

Untuk melihat daftar secret yang ada, dapat menggunakan perintah berikut

xxxxxxxxxx
kubectl get secret --all-namespaces

Ganti --all-namespaces menjadi --namespace=<namespace> untuk melihat daftar secret dalam suatu namespace.

Perbedaan secret dari configmap, selain penggunaanya yang untuk data yang rahasia, secret juga memiliki beberapa tipe berdasarkan untuk apa secret itu digunakan. Terdapat 8 jenis secret:

Untuk membuat sebuah secret, terdapat tiga metode melalui kubectl berdasarkan tipe dari secret tersebut

  1. Metode generic, yaitu metode yang digunakan untuk membuat secret secara general untuk semua tipe. Perintahnya adalah sebagai berikut

    xxxxxxxxxx
    kubectl create secret generic <secret_name> [--namespace <name_space>] [--type <type>] [--from-literal=<key>=<value>] [--from-file=[<key>=]<filename>] [--from-file=<directory>] [--from-env-file=<envfilename>]

  2. Metode docker-registry, yaitu metode yang digunakan untuk membuat secret khusus bertipe kubernetes.io/dockerconfigjson. Perintahnya adalah sebagai berikut

    xxxxxxxxxx
    kubectl create secret docker-registry <secret_name> [--namespace <name_space>] [--docker-server=<docker_server --docker-username=<docker_username> --docker-password=<docker_password> --docker-email=<docker_email>] [--from-file=.dockerconfigjson=<docker_config_file>]

  3. Metode tls, yaitu metode yang digunakan untuk membuat secret khusus bertipe kubernetes.io/tls. Perintahnya adalah sebagai berikut

    xxxxxxxxxx
    kubectl create secret tls <secret_name> --cert=<cert_file> --key=<key_file> [--namespace <name_space>]

Untuk menghapus secret, dapat menggunakan perintah berikut

xxxxxxxxxx
kubectl delete secret <secret_name> [--namespace <namespace>]

Berikut adalah struktur berkas yaml dari secret

xxxxxxxxxx
apiVersion: v1
kind: Secret
immutable: <immutable: boolean>
type: <type: string>
metadata:
  name: <secret_name: string>
  namespace: <namespace: string>
  labels:
    <label_key: string>: <label_value: string>
  annotations:
    <annotation_key: string>: <annotation_value: string>
data:
  <data_key: string>: <data_value: base64>
stringData:
  <string_data_key: string>: <string_data_value: string>

Perlu diperhatikan bahwa data yang dimasukkan dalam parameter data harus di-encode ke dalam base64. Gunakan parameter stringData jika ingin memasukkan data string biasa.
Jika ada key yang sama pada parameter data dan stringData, maka nilai yang dipilih adalah yang dari stringData.

Key dari parameter data pada setiap tipe secret berbeda-beda. Berikut penjelasannya:

  1. Secret tipe Opaque memiliki key bebas sesuai keinginan pengguna.

  2. Secret tipe kubernetes.io/service-account-token memiliki key:

    • namespace, yaitu namespace dari serviceaccount.
    • ca.crt, yaitu certificate authority untuk mengakses cluster.
    • token, yaitu token dari serviceaccount

  3. Secret tipe kubernetes.io/dockercfg memiliki key .dockercfg yang merupakan isi berkas ~/.dockercfg.

  4. Secret tipe kubernetes.io/dockerconfigjson memiliki key .dockerconfigjson yang merupakan isi berkas ~/.docker/config.json.

  5. Secret tipe kubernetes.io/basic-auth memiliki key username dan password.

  6. Secret tipe kubernetes.io/ssh-auth memiliki key ssh-privatekey.

  7. Secret tipe kubernetes.io/tls memiliki key tls.cert yang merupakan isi berkas certificate dan tls.key yang merupakan isi berkas private key.

  8. Secret tipe bootstrap.kubernetes.io/token memiliki key token-id, token-secret, description, expiration, usage-bootstrap-authentication, usage-bootstrap-signing, dan auth-extra-groups.

Secret dapat dimasukkan ke dalam container sebagai environment variable baik seluruh data maupun beberapa. Untuk memasukkan suatu data dari secret ke dalam container, gunakan parameter berikut di dalam struktur dari container

xxxxxxxxxx
env:
  - name: <env_name: string>
    valueFrom:
      secretKeyRef:
        key: <data_key: string>
        name: <secret_name: string>
        optional: <optional: boolean>

Untuk memasukkan seluruh data dari secret ke dalam container, gunakan parameter berikut di dalam struktur dari container

xxxxxxxxxx
envFrom:
  - prefix: <prefix: string>
    secretRef:
      name: <secret_name: string>
      optional: <optional: boolean>

Berikut adalah contoh berkas yaml dari secret

xxxxxxxxxx
apiVersion: v1
kind: Secret
type: `kubernetes.io/basic-auth`
metadata:
  name: admin
  namespace: default
stringData:
  username: admin
  password: 1234

ResourceQuota

ResourceQuota
adalah suatu objek berupa daftar kuota resource yang dapat digunakan oleh suatu kelompok objek (misalnya pod) berdasarkan namespace maupun berdasarkan jenis.

Resourcequota berguna untuk membatasi penggunaan resources oleh pengguna sehingga tidak terlalu boros dalam penggunaan resources.

Untuk melihat daftar resourcequota yang ada, dapat menggunakan perintah berikut

xxxxxxxxxx
kubectl get resourcequota --all-namespaces

Ganti --all-namespaces menjadi --namespace=<namespace> untuk melihat daftar resourcequota dalam suatu namespace.

Untuk menghapus resourcequota, dapat menggunakan perintah berikut

xxxxxxxxxx
kubectl delete resourcequota <resourcequota_name> [--namespace <name_space>]

Berikut adalah struktur berkas yaml dari resourcequota

xxxxxxxxxx
apiVersion: v1
kind: ResourceQuota
metadata:
  name: <resourcequota_name: string>
  namespace: <namespace: string>
  labels:
    <label_key: string>: <label_value: string>
  annotations:
    <annotation_key: string>: <annotation_value: string>
spec:
  hard:
    <resource_key: string>: <resource_value>
  scopes: [<scope: string>]
  scopeSelector:
    matchExpressions:
      - scopeName: <scope_name: string>
        operator: <operator: string>
        values: [<value>]

Kumpulan resource yang bisa digunakan sebagai <resource_key> adalah sebagai berikut

Jika parameter spec.scopes[] atau spec.scopeSelector digunakan, maka resource yang dapat digunakan sebagai <resource_key> adalah limits.cpu, limits.memory, requests.cpu, requests.memory, dan pods.

Kumpulan filter yang bisa digunakan sebagai <scope> dan <scope_name> adalah sebagai berikut:

Untuk parameter spec.scopeSelector.matchExpressions[].operator, nilai yang mungkin adalah:

Berikut adalah contoh berkas yaml dari resourcequota

xxxxxxxxxx
apiVersion: v1
kind: ResourceQuota
metadata:
  name: high-class
spec:
  hard:
    limits.cpu: 32
    limits.memory: 100Gi
    requests.cpu: 32
    requests.memory: 100Gi
    pods: 50

LimitRange

LimitRange
adalah suatu objek berupa pengaturan nilai maksimum, minimum, dan default dari resource yang digunakan pada tiap objek (misalnya container dalam pod) di dalam suatu namespace.

Limitrange berguna untuk mengatur batasan resource yang digunakan pada tiap objek. Berbeda dengan resourcequota yang mengatur total resource pada suatu kelompok objek, limitrange mengatur tiap masing-masing objek.

Untuk melihat daftar limitrange yang ada, dapat menggunakan perintah berikut

xxxxxxxxxx
kubectl get limitrange --all-namespaces

Ganti --all-namespaces menjadi --namespace=<namespace> untuk melihat daftar limitrange dalam suatu namespace.

Untuk menghapus limitrange, dapat menggunakan perintah berikut

xxxxxxxxxx
kubectl delete limitrange <limitrange_name> [--namespace <name_space>]

Berikut adalah struktur berkas yaml dari limitrange

xxxxxxxxxx
apiVersion: v1
kind: LimitRange
metadata:
  name: <limitrange_name: string>
  namespace: <namespace: string>
  labels:
    <label_key: string>: <label_value: string>
  annotations:
    <annotation_key: string>: <annotation_value: string>
spec:
  limits:
    - type: <type: string>
      default: 
        <resource_key: string>: <resource_value>
      defaultRequest: 
        <resource_key: string>: <resource_value>
      max:
        <resource_key: string>: <resource_value>
      min:
        <resource_key: string>: <resource_value>
      maxLimitRequestRatio: 
        <resource_key: string>: <resource_value: number>

Objek yang dapat diatur oleh limitrange adalah sebagai berikut:

  • Pod, dengan <resource_key> yang bisa diatur adalah cpu dan memory.
  • Container, dengan <resource_key> yang bisa diatur adalah cpu dan memory.
  • PersistentVolumeClaim, dengan <resource_key> yang bisa diatur adalah storage (tidak bisa mengatur spec.limits[].default dan spec.limits[].defaultRequests)

Berikut adalah contoh berkas yaml dari limitrange

apiVersion: v1
kind: LimitRange
metadata:
  name: strict
spec:
  limits:
    - type: Container
      default:
        memory: 1Gi
        cpu: 1
      defaultRequests:
        memory: 1Gi
        cpu: 1
      max:
        memory: 50Gi
        cpu: 32
      maxLimitRequestsRatio:
        memory: 2
        cpu: 2
    - type: PersistentVolumeClaim
      max:
        storage: 50Gi
      min:
        storage: 1Gi
