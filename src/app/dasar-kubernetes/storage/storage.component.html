7. Storage

Dalam bab ini, akan dijelaskan mengenai objek Kubernetes yang berhubungan dengan penyimpanan. Suatu pod dapat mount suatu penyimpanan di luar container dengan bantuan objek dari Kubernetes. Terdapat dua jenis objek penyimpanan dari Kubernetes:

  1. Penyimpanan sementara, yaitu objek penyimpanan yang dibuat berbarengan dengan pembuatan pod dan akan dihapus setelah pod hilang.
  2. Penyimpanan terpisah, yaitu objek penyimpanan yang tidak terikat dengan pod dan berdiri sendiri. Maka dari itu, jika suatu pod untuk mount ke objek penyimpanan tersebut, harus membuat objek untuk request untuk terikat dengan objek penyimpanan.

Untuk penyimpanan terpisah, terdapat dua cara pod melakukan request untuk terikat dengan objek penyimpanan:

  1. Dynamic Volume Provisioning, di mana objek penyimpanan akan dibuat berdasarkan apa yang pod request. Objek untuk request akan meminta objek penyimpanan ke objek penyedia penyimpanan dan akan dibuat objek penyimpanan berdasarkan request. Jika objek yang digunakan untuk request hilang, maka objek penyimpanan juga ikut hilang.
  2. Static Volume Provisioning, di mana objek penyimpanan dibuat secara manual oleh penggunan, setelah itu objek untuk request akan terikat dengan objek penyimpanan tersebut.

Volume

Volume
adalah suatu objek yang merepresentasikan suatu penyimpanan dalam pod yang bersifat terikat pada pod yang artinya penyimpanan akan hilang bersamaan dengan hilangnya pod.

Objek volume tidak bisa berdiri sendiri dan selalu berada di dalam pod dan diatur dalam parameter spec.volumes[] dalam berkas yaml dari pod.

Terdapat berbagai tipe dari volume bergantung pada penyimpanan jenis apa yang digunakan. Di sini akan dijelaskan lima tipe volume:

Berikut adalah struktur dari sebuah volume

xxxxxxxxxx
name: <volume_name: string>
<volume_type: string>:
  <volume_type_parameter: string>: <volume_type_parameter_value>
Tipe emptyDir

berikut adalah struktur dari volume tipe emptyDir

xxxxxxxxxx
emptyDir:
  medium: <medium: string>
  sizeLimit: <size_limit: string>
Tipe hostPath

berikut adalah struktur dari volume tipe hostPath

xxxxxxxxxx
hostPath:
  path: <path: string>
  type: <type: string>

Nilai type yang mungkin di antaranya:

  • DirectoryOrCreate, lokasi penyimpanan berupa directory, jika tidak ada dalam node, maka akan dibuat.
  • Directory, lokasi penyimpanan berupa directory dan harus ada dalam node.
  • FileOrCreate, lokasi penyimpanan berupa berkas, jika tidak ada dalam node, maka akan dibuat berkas kosong.
  • File, lokasi penyimpanan berupa berkas dan harus ada dalam node.
Tipe configMap

berikut adalah struktur dari volume tipe configMap

xxxxxxxxxx
configMap:
  name: <configmap_name: string>
  optional: <optional: string>
  defaultMode: <default_mode: string>
  items:
    - key: <data_key: string>
      mode: <mode: string>
      path: <path: string>

Parameter dari items[] adalah:

  • items[].key adalah parameter yang menunjukkan key dari data dalam configmap yang ingin diubah lokasi dan permission berkasnya.
  • items[].mode adalah parameter yang menunjukkan permission untuk key dari data.
  • items[].path adalah parameter yang menunjukkan lokasi lain relatif terhadap lokasi volume untuk key dari data.
Tipe secret

berikut adalah struktur dari volume tipe secret

xxxxxxxxxx
secret:
  name: <secret_name: string>
  optional: <optional: string>
  defaultMode: <default_mode: string>
  items:
    - key: <data_key: string>
      mode: <mode: string>
      path: <path: string>

Parameter dari items[] adalah:

  • items[].key adalah parameter yang menunjukkan key dari data dalam secret yang ingin diubah lokasi dan permission berkasnya.
  • items[].mode adalah parameter yang menunjukkan permission untuk key dari data.
  • items[].path adalah parameter yang menunjukkan lokasi lain relatif terhadap lokasi volume untuk key dari data.

berikut adalah struktur dari volume tipe persistentVolumeClaim

Berikut adalah contoh berkas yaml dari pod yang memiliki volume

xxxxxxxxxx
apiVersion: v1
kind: Pod
metadata:
  name: ubuntu
  namespace: default
  labels:
    app: ubuntu
spec:
  restartPolicy: Always
  containers:
    - name: ubuntu
      image: ubuntu:focal
      command: ["/bin/bash", "-c", "--"]
      volumeMounts:
        - name: engine
          mountPath: /engine
        - name: admin
          mountPath: /administrator
        - name: webconfig
          mountPath: /config
  volumes:
    - name: engine
      hostPath:
        path: /root/engine
        type: Directory
    - name: admin
      secret:
        name: admin
    - name: config
      configMap:
        name: webconfig
        items:
          - key: username.default
            path: username

Dari contoh di atas, berkas yang berisi nilai dengan key username.default akan ada di /config dengan nama berkas username.

PersistentVolume & PersistentVolumeClaim

PersistentVolume
adalah suatu objek yang merepresentasikan suatu penyimpanan yang berdiri sendiri. Suatu pod membutuhkan objek request untuk terikat dengan persistentvolume sehingga dapat mount ke penyimpanan tersebut.

PersistentVolumeClaim
adalah suatu objek yang merepresentasikan objek request untuk terikat dengan penyimpanan. Objek ini yang akan ditambahkan sebagai volume di dalam pod agar pod dapat mengakses penyimpanan tersebut.

Persistentvolume dan persistentvolumeclaim adalah dua objek yang saling berhubungan satu sama lain. Hubungan antara mereka dapat dilihat berdasarkan status dari kedua objek tersebut

Untuk melihat daftar persistentvolume yang ada, dapat menggunakan perintah berikut

xxxxxxxxxx
kubectl get persistentvolume

Persistentvolume bukan objek yang harus berada dalam namespace sehingga tidak ada parameter --all-namespaces maupun --namespace=<namespace>.

Untuk menghapus persistentvolume, dapat menggunakan perintah berikut

xxxxxxxxxx
kubectl delete persistentvolume <persistentvolume_name>

Untuk melihat daftar persistentvolumeclaim yang ada, dapat menggunakan perintah berikut

xxxxxxxxxx
kubectl get persistentvolumeclaim --all-namespaces

Ganti --all-namespaces menjadi --namespace=<namespace> untuk melihat daftar persistentvolumeclaim dalam suatu namespace.

Untuk menghapus persistentvolumeclaim, dapat menggunakan perintah berikut

xxxxxxxxxx
kubectl delete persistentvolumeclaim <persistentvolumeclaim_name> [--namespace <name_space>]

Berikut adalah struktur berkas yaml dari persistentvolume

xxxxxxxxxx
apiVersion: v1
kind: PersistentVolume
metadata:
  name: <persistentvolume_name: string>
  labels:
    <label_key: string>: <label_value: string>
  annotations:
    <annotation_key: string>: <annotation_value: string>
spec:
  accessModes: [<access_mode: string>]
  persistentVolumeReclaimPolicy: <reclaim_policy: string>
  storageClassName: <storage_class_name: string>
  capacity:
    storage: <storage_capacity>
  claimRef:
    name: <claim_name: string>
    namespace: <claim_namespace: string>
  <volume_type: string>:
    <volume_type_parameter: string>: <volume_type_parameter_value>

Parameter spec.claimRef tidak bersifat mengikat karena jika parameter diisi dengan identitas persistentvolumeclaim yang telah terikat dengan persistentvolume lain, maka tidak serta merta ikatan sebelum dilepas. Ada baiknya parameter ini dikosongkan kecuali memang yakin persistentvolumeclaim tersebut belum terikat.

Tipe dari persistentvolume sama dengan tipe dari volume (kecuali configMap, secret, dan persistentVolumeClaim) dengan tambahan satu jenis tipe yaitu local. Tipe local hampir sama dengan tipe hostPath dengan struktur sebagai berikut:

xxxxxxxxxx
local:
  fsType: <fs_type: string>
  path: <path: string>

Perbedaan tipe local dan hostPath adalah lokasi node yang dipilih. Untuk local, lokasi node bersifat tetap mengikuti lokasi di mana persistentvolume dibuat, sedangkan untuk hostPath, lokasi node mengikuti lokasi di mana pod dibuat.

Berikut adalah struktur berkas yaml dari persistentvolumeclaim

xxxxxxxxxx
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: <persistentvolumeclaim_name: string>
  namespace: <namespace: string>
  labels:
    <label_key: string>: <label_value: string>
  annotations:
    <annotation_key: string>: <annotation_value: string>
spec:
  accessModes: [<access_mode: string>]
  storageClassName: <storage_class_name: string>
  volumeName: <volume_name: string>
  resources:
    limits:
      storage: <storage_capacity>
    requests:
      storage: <storage_capacity>
  selector:
    matchLabels:
      <label_key: string>: <label_value: string>
    matchExpressions:
      - key: <label_key: string>
        operator: <operator: string>
        values: [<value: string>]

Berikut adalah cara kerja LabelSelector:

  1. matchLabels bekerja dengan menyesuaikan semua label_key dan label_value dalam metadata.labels dari objek yang dicari. Semuanya harus cocok.
  2. matchExpressions bekerja dengan menyesuaikan metadata.labels dari objek dengan keys, operator, dan values. operator yang mungkin adalah In (memeriksa apakah label_value dari label_key ada di dalam values), NotIn (memeriksa apakah label_value dari label_key tidak ada di dalam values), Exists (memeriksa apakah key ada dalam metadata.labels), dan DoesNotExists (memeriksa apakah key ada dalam metadata.labels)

Berikut adalah proses membentuk ikatan antara persistentvolume dan persistentvolumeclaim:

  1. Jika parameter spec.volumeName pada persistentvolumeclaim diatur, persistentvolumeclaim akan langsung mencari persistentvolume dengan nama tersebut. Tapi, jika persistentvolume tersebut berada di storageclass yang berbeda dengan yang dicari oleh persistentvolumeclaim, ikatan tidak dibentuk, kecuali parameter spec.claimRef pada persistenvolume tersebut mengarah ke persistentvolumeclaim.
  2. Jika parameter spec.volumeName pada persistentvolumeclaim diatur dan spec.claimRef pada persistentvolume diatur, ikatan terjadi jika dan hanya jika spec.volumeName sesuai dengan nama persistentvolume meskipun spec.claimRef mengarah ke persistentvolumeclaim.
  3. Parameter spec.selector tidak bersifat mengikat jika ada parameter spec.volumeName.
  4. Parameter spec.selector tidak bersifat mengikat jika ada parameter spec.claimRef.

Persistentvolume yang telah dilepas ikatannya oleh persistentvolumeclaim statusnya akan menjadi Release dan tidak bisa diikat oleh persistentvolumeclaim lain karena parameter spec.claimRef telah disesuaikan dengan identitas persistentvolumeclaim sebelumnya, untuk mengubah status persistentvolume menjadi Available sehingga bisa diikat oleh persistentvolumeclaim lain, dapat menggunakan perintah berikut

xxxxxxxxxx
kubectl patch persistentvolume <persistentvolume_name> -p '{"spec": {"claimRef": null}}'

Berikut adalah contoh berkas yaml dari persistentvolume dan persistentvolumeclaim

xxxxxxxxxx
apiVersion: v1
kind: PersistentVolume
metadata:
  name: host-volume
spec:
  accessModes: ["ReadWriteMany"]
  persistentVolumeReclaimPolicy: Retain
  storageClassName: ""
  capacity:
    storage: 50Gi
  claimRef:
    name: host-volume-claim
    namespace: default
  hostPath:
    path: /root/engine
    type: Directory
---
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: host-volume-claim
  namespace: default
spec:
  accessModes: ["ReadWriteMany"]
  storageClassName: ""
  resources:
    limits:
      storage: 20Gi
    requests:
      storage: 10Gi
  volumeName: host-volume

StorageClass

StorageClass
adalah suatu objek yang merepresentasikan sebagai penerima request dari persistentvolumeclaim dan membuat persistentvolume jika tidak ada yang bisa terikat dengan persistentvolumeclaim.

Untuk melihat daftar storageclass yang ada, dapat menggunakan perintah berikut

xxxxxxxxxx
kubectl get storageclass

Storageclass bukan objek yang harus berada dalam namespace sehingga tidak ada parameter --all-namespaces maupun --namespace=<namespace>.

Untuk menghapus storageclass, dapat menggunakan perintah berikut

xxxxxxxxxx
kubectl delete storageclass <storageclass_name>

Berikut adalah struktur berkas yaml dari storageclass

xxxxxxxxxx
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: <storageclass_name: string>
  labels:
    <label_key: string>: <label_value: string>
  annotations:
    <annotation_key: string>: <annotation_value: string>
allowVolumeExpansion: <allow_volume_expansion: boolean>
provisioner: <provisioner: string>
parameters:
  <provisioner_parameter: string>: <provisioner_parameter_value: string>
reclaimPolicy: <reclaim_policy: string>
volumeBindingMode: <volume_binding_mode: string>

Untuk provisioner tipe local, sebaiknya volumeBindingMode diatur menjadi WaitForFirstConsumer.

Berikut adalah contoh berkas yaml dari storageclass

apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: local-storage
allowVolumeExpansion: false
provisioner: nfs
parameters:
  archiveOnDelete: false
